# opencode-guardrails

Распространяемая **система уровней строгости подтверждений** для
[opencode](https://opencode.ai). Она заменяет встроенную пару `build`/`plan` одним
переключателем строгости: три primary-агента, которые циклятся по **Tab** — от «спрашивать
обо всём» до «запускать почти что угодно» — плюс несдвигаемый floor, который на любом
уровне держит под охраной секреты и небольшой набор опасных shell-команд.

## Назначение

opencode поставляет двух primary-агентов (`build`, `plan`), которые смешивают *что делает
модель* с *тем, насколько она спрашивает*. Этот проект сводит всё к одной оси — **насколько
строгие подтверждения** — и выдаёт её в виде трёх взаимозаменяемых агентов:

| Агент          | Смысл                                                                    |
|----------------|--------------------------------------------------------------------------|
| `ask`          | Каждое действие спрашивает. Ревью каждого чтения, правки, команды, поиска. |
| `normal`       | Сбалансированный дефолт. Чтение/поиск идут свободно; правки и shell спрашивают. |
| `trust`        | Самый смелый. Почти всё выполняется; секреты и опасный shell под охраной. |

Это обычные primary-агенты opencode, поэтому они попадают в цикл **Tab**. Overlay (см.
[Установку](#установка)) делает `guard/normal` дефолтным и убирает `build`/`plan`, так что
цикл Tab становится ровно этими тремя уровнями. Ни один из них не меняет поведение или
персону модели — они задают **только** уровень подтверждений.

> **Установленные имена.** Установщик кладёт трёх агентов в подкаталог `guard/`
> директории `agent/` opencode, а opencode именует агента по его пути ниже `agent/`.
> Поэтому в цикле Tab они появляются как `guard/ask` / `guard/normal` / `guard/trust`,
> а дефолт — `guard/normal`. Короткие метки `ask` / `normal` / `trust` используются во
> всём этом документе как имена **уровней**. Группировка под одним подкаталогом
> означает, что весь набор удаляется одной командой `rm -rf .../agent/guard` (см.
> [Удаление](#удаление)).

## Уровни

Каждый уровень — это матрица permission'ов opencode (`allow` / `ask` / `deny`) по
инструментам. Catch-all `"*"` задаётся явно на каждом уровне; конкретные инструменты его
переопределяют.

| Инструмент                        | ask    | normal (дефолт) | trust |
|-----------------------------------|:------:|:---------------:|:-----:|
| `"*"` (catch-all)                 | ask    | ask             | allow |
| `read`                            | ask †  | allow †         | allow † |
| `grep` / `glob`                   | ask    | allow ‡         | allow ‡ |
| `edit`                            | ask    | ask             | allow |
| `bash`                            | ask ✦  | ask ✦           | allow ✦ |
| `webfetch` / `websearch`          | ask    | allow           | allow |
| `task`                            | ask    | ask             | ask   |
| `external_directory` / `doom_loop`| ask    | ask             | ask   |

† `read` подчиняется **floor** (ниже): даже там, где `read` = `allow`, жёсткие секреты
принудительно `deny`, а прочие скрытые файлы — `ask`.
‡ permission'ы `grep`/`glob` матчат *поисковый паттерн*, а не пути файлов, поэтому floor к
ним не применяется — см. [Известные ограничения](#известные-ограничения).
✦ `bash` всегда сохраняет best-effort floor опасных команд (ниже).

### Floor

Floor — это фиксированный блок правил, **дописанный последним** в конфиге `read`/`bash`
каждого уровня. opencode резолвит permission'ы по принципу **last-match-wins**, поэтому
дописанный последним floor перебивает `allow` уровня. Блок **побайтово одинаков во всех
трёх файлах агентов** (это гарантирует инвариант-тест).

- **Жёсткие секреты → `deny` (на `read`).** `.env` / `*.env` / `*.env.*`, `*.pem`,
  `*.key`, SSH-ключи `id_rsa` / `id_ed25519` / `id_ecdsa` / `id_dsa`, plaintext-файлы
  учёток `.netrc` / `.pgpass` / `.git-credentials`, `credentials` / `credentials.*`,
  `secrets` / `secrets.*`, а также *содержимое* каталога `credentials/` или `secrets/`
  (каждый со своим вложенным `*/…`-двойником) резолвятся в `deny`. `deny` используется
  вместо `ask`, потому что он **переживает `--auto`** (auto-approve гасит `ask`, но не
  `deny`), и потому что эти файлы редко читаются легитимно. Это **усиливает** встроенную
  обработку `.env` в opencode — чей дефолт `ask`, а не `deny` — до жёсткого `deny`.
  Паттерны **якорные, не substring**, поэтому обычные доки вроде `credentials-guide.md`
  или `secrets-overview.md` *не* попадают под deny. Carve-out для `*.env.example`
  (дописан последним) оставляет шаблоны читаемыми. Поскольку floor побайтово одинаков
  во всех уровнях, и слой `deny`, и этот carve-out действуют даже на `ask`: секреты
  запрещены, а `.env.example` авто-разрешён на любом уровне, так что «спрашивать про
  всё» у `ask` означает «спрашивать про всё, что floor ещё не решил сам».
- **Прочие скрытые файлы → `ask` (на `read`).** Dot-именованные файлы (`.*`, `*/.*`,
  например `.gitignore`, `.eslintrc`) запрашивают подтверждение. Этот слой *условен*:
  `--auto` или сессионное `always`-одобрение могут подавить prompt.
- **Опасный bash → `ask` (best-effort).** Частые прямые формы — `git push*`, `rm *`,
  `git reset --hard*`, `curl … | sh` — плюс best-effort чтение секретов через shell
  (`cat` / `less` / `head` / `tail` файлов `*.env` / `*.env.*` / `*.pem` / `*.key`,
  напр. `cat .env`) — спрашивают даже на `trust`. Это только best-effort: цепочки,
  переменные и обфускация его обходят (`c=cat; $c .env`), и он покрывает четыре
  частых ридера на классах файлов env/pem/key, а не любой ридер и не любое имя
  секрета (см. [Известные ограничения](#известные-ограничения)).

## Модель угроз — только неосторожный агент

**Этот инструмент защищает ровно от одного противника: неосторожного агента (модель A).**

- **A — неосторожный агент (в скоупе).** LLM в ходе обычной работы *случайно* читает
  секрет или запускает деструктивную команду. Без намерения обойти защиту. Floor снижает
  такие случайности.
- **B — атакующий / prompt-injection (вне скоупа).** Враждебная инструкция (в загруженном
  веб-контенте, файле, issue), которая *активно* пытается вынести секреты или навредить и
  будет обфусцировать, чтобы обойти любой паттерн.

**opencode-guardrails — НЕ граница безопасности против B.** Config-паттерны ею быть не
могут: shell Тьюринг-полон, поэтому любой строковый/распарсенный матчер обходится
(`c=cat; $c .env`, base64, `/proc/self/environ`, секреты уже в env процесса, …), а каналы
эксфильтрации безграничны. Permission-движок видит команду, а не намерение. Заявлять
полу-защиту от B было бы хуже, чем ничего — это порождает ложную уверенность.

Полное обоснование:
[docs/adr/0001-threat-model-careless-agent.md](docs/adr/0001-threat-model-careless-agent.md).

## Hardening

Если вам нужна защита от **вредоносных/инъектированных** инструкций (модель B), не
полагайтесь на эти guardrails. Реальную защиту получают **композицией**: запускайте
opencode внутри **OS-sandbox** — Linux namespaces / seccomp / Landlock, контейнер или ВМ —
с allowlist'ом файловой системы и ограниченным сетевым egress. Это отдельный слой, который
действительно сдерживает враждебный процесс; guardrails поверх него снижают *случайные*
ошибки, но сами по себе границей не являются.

## Установка

Требуется рабочая установка opencode (см. [Совместимость](#совместимость)). Склонируйте
репозиторий в любое место и запустите `install.sh` по его пути. В **глобальном** режиме
установщик *копирует* агентов и overlay в конфиг opencode, поэтому место клона не важно и
клон можно удалить после установки (держите его лишь для `git pull` обновлений и
переустановки). В режиме **`--project`** назначение — ваш текущий каталог, см. раздел ниже.

### Основной путь — глобальная установка + overlay через `OPENCODE_CONFIG_DIR`

```sh
./install.sh
```

Это копирует три файла уровней `.md` (`ask.md`/`normal.md`/`trust.md`) в подкаталог
`guard/` глобальной директории агентов opencode
(`${XDG_CONFIG_HOME:-~/.config}/opencode/agent/guard/`, откуда opencode всегда их
грузит — подкаталог `guard/` даёт им имена `guard/ask` / `guard/normal` / `guard/trust`)
и кладёт overlay [`opencode.json`](opencode.json) в отдельную drop-in-директорию
(`${XDG_CONFIG_HOME:-~/.config}/opencode-guardrails/`).

Overlay (`default_agent: guard/normal` + отключение `build`/`plan`) активируется указанием
opencode на эту drop-in-директорию через **`OPENCODE_CONFIG_DIR`**. Скрипт **не может**
экспортировать env-переменную в родительский shell, поэтому `install.sh` вместо этого
**печатает точную команду**:

```sh
export OPENCODE_CONFIG_DIR="${XDG_CONFIG_HOME:-$HOME/.config}/opencode-guardrails"
# сделать постоянным:
echo 'export OPENCODE_CONFIG_DIR="${XDG_CONFIG_HOME:-$HOME/.config}/opencode-guardrails"' >> ~/.bashrc
```

Drop-in-директория — это `${XDG_CONFIG_HOME:-$HOME/.config}/opencode-guardrails`, так что
если у вас задан `XDG_CONFIG_HOME`, путь следует за ним — копируйте **точную команду,
которую печатает `install.sh`**, а не хардкодьте `~/.config`.

opencode *мерджит* слои конфига, а не подменяет их, поэтому это добавляет overlay поверх
вашего глобального конфига, не редактируя ничего вашего.

**Оговорка про precedence.** Overlay попадает в слой *custom*, а порядок слоёв —
`remote → global → custom → project`. Поэтому **project**-`opencode.json`
(`./opencode.json` или `./.opencode/opencode.json`) **переопределяет** overlay: он может
отыграть назад `default_agent` и вернуть `build`/`plan` для этого проекта. В таком проекте
используйте `--project`.

### Без env-переменной — впишите overlay в глобальный конфиг

Не хотите трогать shell-профиль — и хотите, чтобы активация держалась во **всех
терминалах**? Откажитесь от `OPENCODE_CONFIG_DIR` вовсе и впишите две настройки overlay
прямо в **глобальный конфиг opencode**, который opencode читает сам в любом shell.

1. Установите только агентов (грузятся автоматически, переменная не нужна):

   ```sh
   ./install.sh   # копирует ask.md/normal.md/trust.md в ${XDG_CONFIG_HOME:-~/.config}/opencode/agent/guard/
   ```

   Напечатанную команду `OPENCODE_CONFIG_DIR` можно проигнорировать — этот способ не
   использует drop-in-директорию.

2. Добавьте `default_agent` и карту отключения `build`/`plan` в глобальный конфиг
   `${XDG_CONFIG_HOME:-~/.config}/opencode/opencode.json`. opencode также принимает
   `opencode.jsonc` (JSON с комментариями) — если такой файл уже есть, правьте его; если
   нет ни того ни другого, создайте файл:

   ```jsonc
   {
     "$schema": "https://opencode.ai/config.json",
     "default_agent": "guard/normal",
     "agent": {
       "build": { "disable": true },
       "plan": { "disable": true }
     }
   }
   ```

У opencode **нет команды `config set`** — этот файл правится руками. Поскольку это
глобальный слой, который opencode читает всегда, `guard/normal` становится дефолтным, а
`build`/`plan` уходят из Tab-цикла во всех терминалах — без env-переменной и без правки
`.bashrc`. Оговорка про precedence та же: **project**-`opencode.json` всё равно
переопределяет глобальный слой.

### Режим `--project`

Сначала `cd` в проект, который хотите защитить, затем запустите установщик **по его пути** —
`./.opencode/` создаётся относительно вашего текущего каталога (`$PWD`), **а не** каталога
клона:

```sh
cd /path/to/ваш-проект
/path/to/opencode-guardrails/install.sh --project
```

Пишет агентов в `./.opencode/agent/guard/`, а overlay — в `./.opencode/opencode.json`, то есть в
**project-слой**, который выигрывает у global/custom, так что overlay применяется
автоматически для этого проекта **без экспорта env-переменной**. Если project-`opencode.json`
уже существует и отличается, установщик отказывается его затирать и печатает ключи для
ручного мерджа (или перезапустите с `--force`).

### Fallback — только агенты

Если вы никогда не экспортируете `OPENCODE_CONFIG_DIR` (в глобальном режиме), три агента
(`guard/ask`/`guard/normal`/`guard/trust`) всё равно установлены и доступны для выбора — но
**`build`/`plan` остаются в цикле Tab**, а `guard/normal` **не** становится дефолтом. Это
валидная, более лёгкая установка; вы просто переключаете уровень вручную.

`install.sh` идемпотентен, никогда не перезаписывает отличающийся файл без `--force` и
печатает всё вышеописанное (команду активации, оговорку про precedence, fallback) в конце.

## Удаление

Три агента уровней живут под единым подкаталогом `guard/` именно ради того, чтобы весь
набор удалялся одной командой. В **глобальном** режиме:

```sh
rm -rf "${XDG_CONFIG_HOME:-~/.config}/opencode/agent/guard"      # агенты уровней
rm -rf "${XDG_CONFIG_HOME:-~/.config}/opencode-guardrails"       # drop-in overlay
```

Также уберите строку `export OPENCODE_CONFIG_DIR=…` из shell-профиля, если добавляли её.
В режиме **`--project`** удалите `./.opencode/agent/guard` (и overlay
`./.opencode/opencode.json`, если он больше не нужен). `install.sh` печатает точную
команду удаления для вычисленных путей в конце каждого запуска.

## Совместимость

- **Минимально поддерживаемая = проверенная: opencode 1.17.18.**

Всё поведение выше (синтаксис `disable` агентов, `default_agent`, слоирование через
`OPENCODE_CONFIG_DIR`, семантика permission-матчера, директория загрузки `agent/`)
проверено на opencode 1.17.18. Другие версии могут отличаться; матчер и обработка bash
особенно чувствительны к версии (см. ниже).

## Известные ограничения

Честно о том, что floor **не** покрывает:

- **`grep` / `glob` не уважают `read`-deny секретов.** Их permission матчит *поисковый
  паттерн*, а не возвращаемые пути файлов, поэтому `grep` на `normal`/`trust` может вынести
  содержимое секрета. Строгая гарантия `deny` относится **только к `read`**.
- **bash-матчинг — по всей строке, best-effort.** В 1.17.18 bash-permission матчит *всю
  строку команды* (парсер на tree-sitter ещё не портирован). Поэтому цепочки, переменные и
  обфускация обходят паттерны — `c=cat; $c .env`, `echo x | sh`, `foo && rm bar`, всё, что
  не начинается с литерального совпадающего префикса. Это ровно граница модели A, не защита
  от модели B.
- **`--auto` / сессионное `always` гасят слой `ask`.** Они **не** влияют на `deny` —
  жёсткие секреты остаются под запретом — но слой «прочие скрытые файлы → ask» и `ask`-промпты
  опасного bash могут быть подавлены. Поэтому гарантии для ask-слоя условны.
- **Precedence project-конфига.** Как выше, project-`opencode.json` переопределяет
  custom/env-overlay (`project > custom > global`), так что замена дефолтного агента
  гарантирована только при отсутствии конфликтующего project-конфига. Там используйте
  `--project`.

## Структура репозитория

- [`agents/`](agents/) — три написанных руками агента уровней
  ([`ask.md`](agents/ask.md),
  [`normal.md`](agents/normal.md),
  [`trust.md`](agents/trust.md)). Генератора нет; держите floor в синхроне
  вручную.
- [`opencode.json`](opencode.json) — overlay (`default_agent` + отключение
  `build`/`plan`).
- [`install.sh`](install.sh) — установщик для обоих режимов.
- [`docs/adr/0001-threat-model-careless-agent.md`](docs/adr/0001-threat-model-careless-agent.md)
  — решение по модели угроз.
- `tests/` — набор тестов на stdlib `unittest`. Запуск: `python3 -m unittest discover tests`.
